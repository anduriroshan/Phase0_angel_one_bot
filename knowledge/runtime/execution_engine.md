# Execution Engine

> Defines the order lifecycle finite state machine, idempotency guarantees,
> retry semantics, and the boundary between strategy intent and broker action.

**Status:** Phase 1 (planned). No order routing exists in Phase 0.

---

## Role in the System

The execution engine is the **only** component that talks to the broker order API. Strategies do not place orders. Risk does not place orders. Only the execution engine.

```
Strategy ──[SignalEvent]──▶ Risk Engine ──[ApprovedOrder]──▶ Execution Engine ──▶ Broker
                                                                    │
                                                                    └──[OrderEvent / FillEvent]──▶ Event Bus
```

This isolation is structural. See [adr/ADR-006-execution-engine-isolation.md](../adr/ADR-006-execution-engine-isolation.md) and design principles 6, 7, 8 in [vision/design_principles.md](../vision/design_principles.md#architecture-principles).

## Order Lifecycle FSM

Every order is a state machine with these states and transitions:

```
                    ┌────────────────────────────────────┐
                    │                                     ▼
   NEW ──▶ SUBMITTED ──▶ ACKNOWLEDGED ──▶ PARTIALLY_FILLED ──▶ FILLED
              │                │                    │              │
              │                │                    │              │
              ▼                ▼                    ▼              │
            REJECTED        CANCELLED ◀─────────────┘              │
              │                │                                    │
              └────────────────┴────────────────────────────────────┘
                                          │
                                          ▼
                                      TERMINAL
```

| State | Meaning | Next states |
|---|---|---|
| **NEW** | Order created in our process; not yet sent | SUBMITTED, REJECTED (pre-trade risk fail) |
| **SUBMITTED** | HTTP/WS frame sent to broker; awaiting ack | ACKNOWLEDGED, REJECTED, (timeout → STALE) |
| **ACKNOWLEDGED** | Broker confirmed receipt; order is in their book | PARTIALLY_FILLED, FILLED, CANCELLED, REJECTED |
| **PARTIALLY_FILLED** | Some quantity matched; remainder still resting | PARTIALLY_FILLED (more fills), FILLED, CANCELLED |
| **FILLED** | Full quantity matched | TERMINAL |
| **CANCELLED** | Withdrawn (by us or by broker / exchange end-of-day) | TERMINAL |
| **REJECTED** | Broker refused (margin, validation, freeze qty) | TERMINAL |

**Terminal states are absorbing.** Once an order is FILLED / CANCELLED / REJECTED, no further events for that order can change its state. Late events arriving for terminal orders are logged and dropped.

**Stale handling**: SUBMITTED orders that don't get an ack within `ACK_TIMEOUT_MS` (default: 2000ms) transition to a `PROBABLY_LOST` state. The execution engine queries broker for status before assuming cancellation. Unilateral assumption that a stale order didn't reach the exchange is forbidden — it leads to phantom positions.

## Order Identity & Idempotency

Every order has two identifiers:

| ID | Source | Purpose |
|---|---|---|
| `client_order_id` | Generated by us, before submission | Idempotency. Survives restarts. |
| `broker_order_id` | Returned by broker on ACK | Reference for cancel/modify. Lifetime: until terminal state. |

**`client_order_id` is the system's identity.** It is constructed deterministically from `(strategy_id, signal_id, attempt_no)` so that retries are recognized as retries by the broker, not as new orders. This is the single most important invariant — losing it produces duplicate orders, which can blow up positions.

```rust
// Deterministic, idempotent construction
fn client_order_id(strategy: &str, signal: &SignalId, attempt: u8) -> String {
    format!("{strategy}-{signal}-{attempt:02}")
}
```

Angel One's order placement endpoint accepts a client-supplied identifier. Resubmitting with the same `client_order_id` produces an idempotent response (same `broker_order_id`) — verify per current Angel One docs before relying on this. If the broker does not support idempotency keys, the execution engine maintains a local persistent journal of `(client_order_id → broker_order_id)` so it never resends an order whose ack was already observed in a previous session.

## Retry Semantics

Retries are restricted to **transport failures** (TCP reset, HTTP 5xx, timeout). Application-level rejections (margin shortfall, validation error, freeze qty exceeded) are **never retried** — they are emitted as `OrderRejected` and the strategy decides what to do.

| Failure | Retry? | Backoff |
|---|---|---|
| TCP / TLS error | Yes | Exponential, max 3 attempts, max 5s total |
| HTTP 5xx | Yes | Same as above |
| HTTP 4xx (validation) | No | Single rejection, propagate to strategy |
| Broker-level reject (margin, freeze) | No | Single rejection, propagate to strategy |
| Timeout (no response) | **Query first**, then retry only if broker confirms not received | — |

The "query first on timeout" rule is critical. If we don't know whether the order reached the broker, retrying may double-submit. We **always** query the broker for current order state before retrying, using the `client_order_id`.

## Order Types Supported (Phase 1)

For NSE F&O via Angel One:

| Type | Use case | Risk profile |
|---|---|---|
| **LIMIT** | Default; passive entry, capture spread | No slippage, no guaranteed fill |
| **MARKET** | Emergency exits, circuit breaker panic | Slippage proportional to depth |
| **STOPLOSS_LIMIT** | Risk management with price ceiling on exit | Skips if gap-down through trigger |
| **STOPLOSS_MARKET** | Guaranteed exit on trigger | Worst-case slippage |

Bracket orders, iceberg orders, and basket orders are out of scope for Phase 1.

## Pre-Send Risk Gate

Every order, before SUBMITTED, passes through the risk engine:

```
SignalEvent ──▶ build_order(signal) ──▶ Risk.check_pre_trade(order) ──▶ Submit
                                                  │
                                                  └──▶ Reject ──▶ OrderRejected (pre-trade)
```

Pre-trade checks are listed in [runtime/risk_engine.md](risk_engine.md#pre-trade-checks-planned). These are enforced **inside** the execution engine, not by callers. A strategy cannot disable risk checks. See [vision/system_philosophy.md](../vision/system_philosophy.md#7-risk-is-infrastructure-not-application-code).

## Events Emitted

Every state transition emits an event onto the event bus:

```rust
enum OrderLifecycleEvent {
    OrderSubmitted(OrderEvent),        // NEW → SUBMITTED
    OrderAcknowledged(OrderEvent),     // SUBMITTED → ACKNOWLEDGED
    OrderRejected(RejectionEvent),     // → REJECTED
    OrderPartiallyFilled(FillEvent),   // → PARTIALLY_FILLED
    OrderFilled(FillEvent),            // → FILLED
    OrderCancelled(CancelEvent),       // → CANCELLED
    OrderStale(OrderEvent),            // SUBMITTED + timeout
}
```

Schema details and version discipline: [standards/event_contracts.md](../standards/event_contracts.md).

The event log is the single source of truth — including for replay. See [runtime/replay_engine.md](replay_engine.md) for how the same FSM is driven by recorded events during backtests.

## Process Boundary

The execution engine is a **separate Tokio task** within the trading process (not a separate OS process — that distinction is reserved for the circuit breaker, see [adr/ADR-003-circuit-breaker-isolation.md](../adr/ADR-003-circuit-breaker-isolation.md)). It owns:

- An inbound channel of `ApprovedOrder` from the risk engine
- An outbound channel of `OrderLifecycleEvent` to the event bus
- An HTTP client to the broker
- A persistent local journal of `(client_order_id → broker_order_id, last_known_state)` for restart recovery

It does not own:
- The order book (read via shared `OrderBookCache`)
- Position / PnL state (the state engine owns this; execution emits events)
- Strategy logic

NautilusTrader's `execution` crate is the reference. It implements a far richer model (order emulator, matching engine for backtests, reconciliation), but the FSM and event taxonomy translate directly:

| Order matching for backtests | `knowledge/nautilus_trader/crates/execution/src/matching_engine/` |
| Reconciliation (state recovery on restart) | `knowledge/nautilus_trader/crates/execution/src/reconciliation/` |
| Pre-trade protection checks | `knowledge/nautilus_trader/crates/execution/src/protection.rs` |
| Trailing stop logic | `knowledge/nautilus_trader/crates/execution/src/trailing.rs` |

**What to steal:** the FSM transitions, the `OrderManager` in-flight map, the reconciliation pattern (on startup, query broker for all open orders before accepting new signals).

**What to skip in Phase 1:** order emulator (we don't simulate; we go to broker), trailing stops (defer to Phase 2 — can be implemented above the engine as a strategy concern).

## Restart Recovery

On startup, the execution engine performs reconciliation **before accepting any new signals**:

1. Load the local journal of in-flight `client_order_id`s.
2. Query the broker for the current state of each.
3. For any that the broker reports as filled / cancelled / rejected, emit the corresponding terminal event.
4. For any still working, restore them to the in-flight map.
5. For any in our journal that the broker has no record of, emit `OrderStale` and require manual intervention.

Until reconciliation completes, the inbound channel of `ApprovedOrder` is paused. This prevents double-submission across restarts.
